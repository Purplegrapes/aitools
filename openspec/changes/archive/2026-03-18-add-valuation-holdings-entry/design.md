## Context

当前实现已经明确了新的产品方向：

- “我的持仓”页只保留一个“同步持仓”入口。
- “同步持仓”页只负责选择同步方式，不承载实际录入表单。
- 手动录入和上传截图是两个独立页面，不在同页切换。
- 选择“上传截图”后应立即唤起系统图片选择，识别完成后再进入确认页。
- 确认页以简单直观为优先，只展示识别成功的基金，失败项直接过滤。

这意味着最初 artifacts 里“添加持仓页内切换双入口”“识别失败项保留并人工修复”“支持基金重新匹配”等方案已经不再成立，需要更新为更轻、更直接的同步持仓流程。

## Goals / Non-Goals

**Goals**

- 让“我的持仓”页入口更单一，用户只需要理解“同步持仓”这一件事。
- 保持手动录入和上传截图为两个独立页面，减少页面内来回切换。
- 让上传截图路径更短：选择方式后直接选图，识别完成后直接进入确认页。
- 让确认页尽量简单，只展示成功识别的基金，并支持少量必要操作。
- 保证两条录入路径最终仍然共用同一套持仓保存模型。

**Non-Goals**

- 不实现 OCR 失败项的人工修复工作台。
- 不在确认页支持基金重新匹配。
- 不引入 Excel / CSV 导入。
- 不实现识别历史或批次管理。

## Decisions

### 1. 用“同步持仓”替代“添加持仓/上传截图”多入口

决定：

- “我的持仓”页和空状态都只保留“同步持仓”入口。
- 有持仓时按钮放在页头；空状态下按钮放在空态卡片里。

原因：

- 用户目标是把外部持仓同步进来，而不是区分入口。
- 单入口比多个并列按钮更利于理解，也更节制。

### 2. 手动录入和上传截图保持为独立页面

决定：

- `holdings-add.vue` 只负责手动录入。
- `holdings-upload.vue` 只负责识别后的确认导入。
- `holdings-sync.vue` 只负责方式选择。

原因：

- 用户已明确不希望在一个页面里来回切换。
- 独立页面职责更清晰，也便于后续分别优化。

### 3. 上传截图在方式页直接唤起系统选图

决定：

- 点击“上传截图”时，先在 `holdings-sync.vue` 调用 `uni.chooseImage`。
- 识别成功后通过前端临时会话状态把草稿传到 `holdings-upload.vue`。

原因：

- 避免先跳转到一个空上传页，再进行选图。
- 用户体感更接近“点击上传 -> 立刻开始上传”。

### 4. 确认页只展示识别成功的基金

决定：

- OCR 返回后，前端先做状态归一化，再过滤，只保留 `ready` 项。
- 若没有任何成功项，则直接进入空结果状态。

原因：

- 用户已明确希望失败项直接过滤，不进入确认页处理。
- 这会让确认页更短、更直观，减少复杂异常交互。

### 5. 确认页采用表格式清单，不用卡片堆叠

决定：

- 确认页使用清单式表格，列为：基金、持有金额、持有收益、操作。
- 基金名称只读，单行截断；金额和收益可编辑；操作只保留删除图标。

原因：

- 用户要“简单直观”，表格比多张卡片更适合扫读和快速确认。
- 在只保留成功识别项后，表格足够表达信息，不需要状态卡片。

## Risks / Trade-offs

- [失败项被直接过滤] → 用户看不到失败原因，但确认页会更简单；这符合当前产品取舍。
- [同步页先识别再跳转] → 需要一个临时会话状态承接识别结果；已用前端 session 解决。
- [确认页不支持重新匹配] → 若 OCR 匹配率较低，会降低导入量；但当前方向优先简化交互。

## Migration Plan

1. 将“我的持仓”页入口统一为“同步持仓”。
2. 新增“同步持仓”方式选择页。
3. 保持手动录入页独立，继续沿用现有保存模型。
4. 在同步方式页直接触发截图选择和识别。
5. 用前端临时会话把成功识别项交给确认页。
6. 将确认页改成表格式，只展示成功识别项，并支持删除与确认导入。

## Open Questions

- 后端 OCR 是否最终也应只返回可导入项，还是仍返回完整原始结果供前端过滤？
- 若未来想补回失败项处理，是否应新增“查看未识别项”入口而不是直接塞回确认页？
