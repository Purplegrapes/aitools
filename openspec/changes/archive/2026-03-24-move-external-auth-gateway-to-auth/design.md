## Context

当前小程序外部访问链路由 `src/subPages/tamp/index.vue` 负责，但它已经不是单纯的 TAMP 业务页，而是统一入口网关：识别 `from` 来源、执行 `transferH5Ticket` 换 token、补齐用户态、在成功后直接跳转 `referer` 指向的完整网站地址，并在失败时按来源分流。与此同时，`auth` 子包已经拥有登录页、登录态写入工具和登录守卫，认证相关能力实际上被拆散在 `auth` 与 `tamp` 两处。

这造成三个具体问题：
- 目录职责不一致，后续维护者很难第一眼判断“外部认证入口”应该去哪里修改。
- `tamp` 业务页与鉴权入口耦合，任何调整中转逻辑都要穿过业务目录。
- 路由、工具函数、store 命名都默认依附于 `tamp`，不利于后续继续扩展更多外部接入入口。

本次设计目标是把“外部认证中转”明确收口到 `auth` 模块，同时通过兼容别名保证现有外链和外部配置短期不失效。

## Goals / Non-Goals

**Goals:**
- 让 `auth` 成为外部来源进入应用时的唯一认证中转归属目录。
- 保持现有小程序 `transferH5Ticket` 换 token、用户信息回填、来源回跳等行为语义不变。
- 为旧的 `/subPages/tamp/index` 提供可控的兼容转发，降低迁移风险。
- 收敛中转相关工具与类型，让未来新增外部入口时优先复用 `auth` 模块。

**Non-Goals:**
- 不改造后端认证接口、参数结构或 token 持久化契约。
- 不在本次提案中重命名 `tampStore` 或一次性移动全部 TAMP API。
- 不调整营销页、demo 页等 TAMP 业务页面的产品行为。

## Decisions

### 1) 在 `auth` 子包新增正式网关页，承接现有中转职责

- 决策：新增 `src/subPages/auth/gateway.vue`（命名可在实现时微调，但必须落在 `auth` 子包），把当前 `tamp/index.vue` 的入口逻辑迁入该页面。
- 原因：路由层面最能体现职责边界，单独页面能让“登录页”和“中转页”都留在 `auth` 目录中，便于维护者理解。
- 备选方案：保留文件路径不变，仅抽公共逻辑到 `auth`；这样实现成本更低，但目录语义仍然是错位的，无法解决你指出的核心问题。

### 2) 保留 `tamp/index` 作为兼容别名，而不是立即删除

- 决策：迁移后保留一个极薄的兼容入口，例如让 `src/subPages/tamp/index.vue` 仅做参数透传并 `router.replace('/subPages/auth/gateway', query)`。
- 原因：外部小程序/H5 很可能已经配置了 `/subPages/tamp/index`，提案阶段不应假设所有接入方能同步切换。
- 备选方案：直接删除旧入口并要求外部统一改链；风险高，且会把目录重构升级成对外断链变更。

### 3) 中转工具按“认证归 auth、业务归 tamp”重分配

- 决策：
  - `detectAccessMode`、referer 解析、旧入口参数归一化等纯鉴权入口能力迁入 `src/subPages/auth/utils/**`
  - `handleExternalRedirect` 如仍只依赖来源与登录地址，也迁入 `auth/utils/**`
  - TAMP 业务专属 API 与业务页留在 `tamp/**`
- 原因：以职责而不是历史目录做归类，避免 `auth` 页面继续反向依赖 `tamp/utils`。
- 备选方案：只移动页面，不移动工具；短期改动少，但会形成新的跨目录反向依赖。

### 4) 认证接口复用现有实现，API 迁移保持渐进

- 决策：本次实现优先复用现有 `transferH5Ticket` 换 token、`getCurrentAuthUser` 等 Method；若需要消除 `auth` 对 `tamp/api` 的依赖，可在实现阶段增加 `auth/api` 的再导出层或迁移到更中性的认证 API 模块。
- 原因：提案目标是职责迁移，不是一次性重做 API 分层。先稳定路由和目录归属，再决定 API 最终归位。
- 备选方案：同步彻底迁移所有认证 API 到 `auth/api`；更“干净”，但改动面更大，容易把本次任务扩成大重构。

### 5) 路由守卫与链接构造统一指向新网关

- 决策：`src/router/index.ts` 中所有“外部小程序先中转后落页”的逻辑统一改为新 `auth` 网关路径；同时提供统一的网关目标构造函数，避免字符串散落。
- 原因：真正的归属迁移不能只改文件位置，还要让新入口成为唯一规范写法。
- 备选方案：仅在旧页内跳转到新页；这会让守卫和新接入继续感知旧 `tamp` 路径，职责边界依然模糊。

### 6) 中转成功后直接跳转 referer，不再做工具页白名单和二跳

- 决策：外部链路完成 `transferH5Ticket` 换票后直接跳转 `referer`，`referer` 可为完整网站 URL，不再要求它是当前项目内的工具页，也不再额外绕行 `subPages/tools/entry`。
- 原因：当前接入方提供的 `referer` 本身就是完整落地地址，中转页职责应聚焦在换票和放行，不额外叠加业务页识别。
- 备选方案：继续保留工具页白名单和二跳；虽然更保守，但会直接阻断真实接入链路。

### 7) 网关迁移不再透传 `appId`

- 决策：新网关和兼容别名在参数透传时不再包含 `appId`，并且仅小程序中转链路要求 `transferH5Ticket` 与 `referer` 为必传字段，`from`、`loginUrl`、`shopId` 等字段均为可选；H5 不走中转页。
- 原因：`appId` 不属于当前认证中转页完成登录、回跳和 tools 二跳所必需的参数，继续透传只会增加心智负担和兼容面。
- 备选方案：继续兼容保留 `appId`；实现上更宽松，但会把已经确认不需要的参数继续固化到新入口契约里。

## Risks / Trade-offs

- [旧外链仍指向 `/subPages/tamp/index`] → 保留兼容别名页，并在文档中声明新标准入口，后续逐步下线旧链路。
- [`auth` 页面继续依赖 `tamp/api`，形成过渡期反向依赖] → 允许过渡期存在，再在后续小步变更中把认证 API 迁到中性目录。
- [直跳完整 URL 带来错误目标地址风险] → 保持 `referer` 必传，并在实现中仅做解码与空值判定，不再混入项目内路径白名单。
- [兼容页实现不当导致重复执行认证流程] → 兼容页只做参数透传和单次 replace，不执行任何鉴权逻辑。
- [开发者继续误用旧入口] → 提供统一入口构造函数，并在 proposal/tasks 中明确把调用点迁到新路径。

## Migration Plan

1. 新增 `auth` 网关页，并把 `tamp/index.vue` 的中转逻辑迁移进去。
2. 将中转页依赖的来源识别、referer 解析、回跳逻辑迁入 `auth` 侧公共工具。
3. 更新路由守卫和任何统一入口构造逻辑，默认跳转到新的 `auth` 网关路径。
4. 把 `tamp/index.vue` 改造成兼容别名页，仅做参数透传和重定向。
5. 验证三条主链路：小程序外部进入并直跳完整 referer、内部未登录跳转、旧 `tamp` 入口兼容跳转。
6. 在完成外部配置切换后，再评估是否移除兼容别名和重命名 `tampStore`。

回滚策略：
- 若迁移后发现外部接入异常，可临时把守卫重新指回 `/subPages/tamp/index`，兼容页本身不会阻碍回滚。
- 因为后端接口和 token 契约不变，回滚主要是前端路由与页面入口层面的切换。

## Open Questions

- 新网关路由命名是否确定为 `/subPages/auth/gateway`，还是沿用 `/subPages/auth/index` 这类更短路径。
- 兼容别名保留多久，是否需要在页面或日志中对旧入口打点，便于后续确认下线窗口。
- `tampStore` 是否在本次实现中保留命名不变，还是顺带抽成更中性的 `externalEntryStore`；当前建议先不动，减少 blast radius。
