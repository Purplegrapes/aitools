## Context

当前 valuation-tool 已具备首页、自选、详情等面向单只基金的轻量工具体验，但“持有”入口仍是预留态，无法帮助用户从“我的钱现在怎么样”这个问题出发理解自己的组合表现。用户画像明确偏向普通基金投资者，页面需要突出收益反馈、陪伴感和结论性阅读体验，而不是证券终端式的密集数据面板。

本次变更需要同时设计一组完整的高保真页面和对应的组件规范，并让这些页面可以在 H5 / 小程序移动端中稳定落地。约束包括：遵守项目的 UnoCSS / rpx / 语义色规范、最大标题不超过 36rpx、交互流程轻量、文案克制且不得越界成投资建议。

## Goals / Non-Goals

**Goals:**
- 提供“我的持仓”完整首版信息架构，包括空状态、添加、总览、编辑、上传预留、骨架屏、数据缺失态。
- 让用户在 3 秒内看清组合累计收益、累计收益率和今日盈亏参考。
- 让用户快速定位贡献最大、拖累最大和值得继续关注的基金。
- 提供足够轻的录入和编辑流程，支持手动添加并预留上传入口。
- 沉淀适合项目的高保真组件规范，确保后续实现一致。
- 将“我的持仓”接入 valuation-tool 首页的底部入口栏。

**Non-Goals:**
- 不做证券账户级别的成交流水、分红、现金管理、税务或复杂资产分析。
- 不做重度记账、复杂图表看板或企业后台式批量管理工具。
- 首版不强制实现真实 Excel/CSV 导入解析，只提供稳定的上传入口和占位流程。
- 不输出明确买卖建议，不提供收益承诺或交易导向文案。

## Decisions

### 1. 将“我的持仓”设计为 valuation-tool 下的独立模块，而不是自选页的扩展
- 决策：持仓页面集作为 `valuation-tool` 内独立子模块存在，首页底部入口直接进入持仓总览或空态页。
- 原因：自选回答“我关注什么”，持仓回答“我已经持有什么、赚亏如何”，两者心智模型不同。独立模块更容易承载组合摘要、录入流程和状态页。
- 备选方案：
  - 直接在自选页上叠加持仓字段：会混淆“关注”和“持有”，页面会迅速变重。
  - 单独放到全局主包：会削弱与现有 valuation-tool 体验的连续性。

### 2. 总览页采用“强摘要 + 观察建议 + 卡片化持仓列表”的轻金融结构
- 决策：总览页顶部用一张资金状态摘要卡承载累计收益、累计收益率、今日盈亏、总市值和持仓数量；中间放一张观察建议卡；下方用卡片式持仓列表展示每只基金状态。
- 原因：用户的第一问题是“整体赚亏”，第二问题是“今天发生了什么”，第三问题才是“单只基金如何”。这种阅读顺序最符合需求中的 3 秒认知目标。
- 备选方案：
  - 全表格化持仓页：虽然专业，但会显得压迫且难以形成收益反馈重心。
  - 证券面板式大数字 + 多图表：超过用户心智负荷，也不符合陪伴感目标。

### 3. 表单流程保持单页轻录入，并把“搜索基金”放在表单首要位置
- 决策：添加持仓页只保留搜索基金、持有份额、成本净值和底部保存操作；编辑页沿用同一表单骨架，并通过弹层或独立页展示删除动作。
- 原因：用户录入的关键摩擦点只有“找到基金”和“填两个数字”。流程越短，越容易建立首次持仓。
- 备选方案：
  - 多步向导：对只有三项关键数据的首版来说过重。
  - 后台表单式长页面：不符合移动端轻量工具定位。

### 4. 上传持仓作为首版预留能力，但设计上仍给出可信的导入流程骨架
- 决策：上传页提供 Excel / CSV 支持说明、文件入口、模板提醒和导入结果预留态；如果首版后端未就绪，也能以说明卡和按钮结构稳定落地。
- 原因：用户确实有批量导入预期，但当前需求允许首版预留。先把流程感建立起来，避免后续整体信息架构重来。

### 5. 数据模型以“持仓记录 + 实时估值聚合”为核心，估值缺失时局部降级
- 决策：每条持仓记录至少包含基金代码、基金名称、持有份额、成本净值；页面总览通过当前估值和持仓份额计算累计收益、累计收益率、今日盈亏和单只贡献。若实时估值缺失，保留累计收益/持仓信息，今日盈亏和实时估值展示缺失态。
- 原因：这能保证用户即使在实时估值暂不可用时，也不会误以为整页报错。
- 备选方案：
  - 整页统一失败：会损害可信度，与需求明确要求相悖。

### 6. 组件规范优先沉淀到可复用的高保真 UI 组件集合
- 决策：明确产出 PortfolioSummaryCard、PositionInsightCard、PositionFundCard、EmptyPortfolioState、AddPositionForm、SearchFundInput、EditPositionSheet、UploadPositionCard、DataUnavailableCard、SkeletonBlock、BottomActionBar 等组件，所有页面围绕这些组件拼装。
- 原因：这次需求本质上既是功能模块设计，也是后续实现蓝图。组件先定义清楚，后续编码时不会退化成零散页面。

## Risks / Trade-offs

- [组合收益计算字段依赖较多] → 先在 spec 中明确最小必需字段，首版可用 mock / fallback 填补，避免卡死实现。
- [上传功能首版可能只有预留态] → 在设计中明确这是“可选 / 预留”，并通过说明文案降低用户预期。
- [持仓总览信息过多容易变成重面板] → 控制信息顺序，只让累计收益、累计收益率、今日盈亏成为绝对主信息，其余作为次级指标。
- [收益相关红绿使用过度会增加压迫感] → 红绿只用于数值和状态标签，主体背景和卡片仍采用轻中性色与蓝色信任色。
- [新增模块会让 valuation-tool 路由继续膨胀] → 通过独立 capability 和子页面分层控制模块边界，并保留组件级复用。

## Migration Plan

- 第一步：在 OpenSpec 中新增 `my-portfolio` capability，并同步修改 `etf-valuation-tool` 首页 requirement，确保需求边界清晰。
- 第二步：实现时新增持仓页面和组件目录，并将首页“持有”入口从占位页升级为正式模块入口。
- 第三步：先用 mock / 本地状态验证持仓总览、录入和编辑流程，再逐步接入真实 API。
- 回滚策略：若持仓模块未完成，可保留首页“持有”入口的占位页，不影响现有估值工具首页、自选页和详情页。

## Open Questions

- 首版真实 API 是否由估值接口直接扩展，还是单独定义持仓聚合接口，目前仍需后续实现阶段确认。
- “更值得关注哪只基金”的判定规则是基于绝对盈亏、今日波动还是观察建议规则，当前设计先以轻量观察建议文案承载。
- 上传持仓是否需要模板下载和导入预览，首版先按预留态设计，后续可按接口能力收敛。
