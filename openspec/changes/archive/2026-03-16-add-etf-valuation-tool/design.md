## Context

当前仓库中的 `src/subPages/etf/index.vue` 是专业向 ETF 估值表，深色、信息密度高，和 `src/subPages/etf/docs/prd.md` 中“基金翻译器”的新手产品定位不一致。文档已经明确了 V1 只需要首页、搜索结果页、基金详情页三段主链路，以及 4 个最小可用接口；参考站点 `https://eagle-lake-83678538.paico.site` 则给出了可还原的浅色卡片式移动端视觉基线。

这次设计的核心不是替换旧 ETF 能力，而是在 `subPages/etf` 下补出一套独立的新工具页流，保持现有项目技术栈和规范：

- Vue 3 + `<script setup lang="ts">`
- Alova `useRequest` 处理接口请求与 loading/error/data
- UnoCSS 作为首选样式方案，颜色与字号遵守项目 token 约束
- `wd-` 组件与全局反馈组件作为基础交互能力

## Goals / Non-Goals

**Goals:**

- 在 ETF 子包中新增面向新手用户的三页工具流：首页、搜索结果页、详情页。
- 页面结构、内容层级与视觉语义对齐参考站点，优先还原其信息组织、卡片布局与轻量情绪感。
- 将 PRD 与 API 文档映射为可实现的前端模块边界，包括页面路由、API 封装、状态模型、兜底策略。
- 明确与现有 ETF 专业页的并存方式，降低实现期对老功能的扰动。

**Non-Goals:**

- 不重构现有 ETF 估值表、基金简况、业绩等专业页。
- 不在本次设计中扩展登录、自选、交易、对比、复杂图表。
- 不新增后端协议，只消费文档中已经定义的 4 个 MVP 接口。
- 不追求像素级复刻站点动画，只要求关键视觉结构、节奏和状态体验一致。

## Decisions

### 1. 新工具采用独立页面流，而不是塞入现有 `etf/index.vue`

推荐方案：在 `src/subPages/etf` 下新增独立入口页、结果页及可复用组件，和当前专业版 ETF 页并存。

原因：

- 现有 `etf/index.vue` 已承担登录、行情、估值表、表格滚动等重逻辑，继续叠加新手工具会让职责混杂。
- PRD 的产品调性与现有页面差异很大，单独页面流更容易控制信息密度与视觉风格。
- 独立路由更方便后续由外部活动页、内容页或 tab 入口直接跳入。

备选方案：

- 在现有 ETF 首页增加 tab 切换模式。缺点是状态耦合重、代码复杂度高、视觉风格冲突明显。

### 2. 按“页面容器 + 业务卡片组件 + API 模块”组织实现

推荐方案：

- 页面层只负责路由参数、请求编排和状态切换。
- 首页拆分为搜索入口、市场情绪卡、热搜榜单、新手说明卡等组件。
- 结果页与详情页分别拆出结果项卡片、身份卡、今日表现卡、事实卡、风险提醒卡等组件。
- API 独立放到 `src/subPages/etf/api/valuationTool.ts` 或同级新模块，避免继续膨胀现有 `api/index.ts`。

原因：

- 符合项目的 Vue 组件边界要求，便于测试和复用。
- 页面结构天然对应参考站点中的多个卡片块，组件化后更好保持 UI 一致性。

备选方案：

- 三个页面都写成单文件大组件。缺点是难维护，状态分支和卡片结构会快速失控。

### 3. 数据状态采用显式枚举，和接口 `status` 对齐

推荐方案：

- 搜索页维护 `idle | loading | success | empty | error`。
- 详情页维护接口定义中的 `ok | not_found | missing_value | loading`，并增加网络错误兜底态。
- 通过类型定义和计算属性统一把接口字段映射为展示文案、标签与徽标颜色。

原因：

- `api.md` 已经给出明确状态契约，沿用该契约可以减少页面判断散落。
- 明确状态枚举后，可以直接驱动 skeleton、空态、缺值态与免责声明展示。

备选方案：

- 在模板里直接基于字段是否存在分支判断。缺点是模板复杂、状态语义不稳定。

### 4. UI 还原以“结构和节奏优先，样式 token 落地”为原则

推荐方案：

- 还原参考站点的浅灰背景、白色卡片、圆角、弱阴影、轻标签、红绿数值和信息提示条。
- 使用项目已有语义色 token；若现有 token 不足，先补充到 `uno.config.ts`，避免业务页面散落十六进制颜色。
- 文案字号维持 `text-sm` 到 `text-lg` 为主，强调值和主标题才提升一级。

原因：

- 满足“按项目规范还原 UI”的要求，同时不破坏全局样式规范。
- 参考站点本身是典型移动端卡片产品，结构还原比逐像素追颜色更关键。

备选方案：

- 直接复制参考站点的任意色值和类名。缺点是与项目 token 体系冲突，后续维护成本高。

### 5. 首期通过 mock 兼容真实接口，保证 apply 阶段可分步交付

推荐方案：

- API 模块先按文档定义好请求函数和类型。
- 如果后端未就绪，页面允许通过本地 mock/默认数据跑通静态态、空态和缺值态。
- 所有 mock 仅作为降级或开发占位，不改变最终 API 契约。

原因：

- 文档清晰但后端可用性未知，这样能降低实现期被接口阻塞的风险。

备选方案：

- 强依赖真实接口后再开发。缺点是实现链路被后端阻塞，UI 还原无法先行验证。

## Risks / Trade-offs

- [新旧 ETF 页面命名和入口混淆] → 在设计与任务中明确新工具是独立路由/入口，避免覆盖现有 `etf` 页面。
- [参考站点只有静态演示，未暴露所有交互细节] → 以 PRD 和 API 文档为主约束，以站点做结构与视觉基线，不推断额外功能。
- [后端接口暂未接通] → 页面状态机先支持 mock 和兜底说明，确保 UI 与交互可以独立验收。
- [UI 还原与项目 token 体系有偏差] → 优先复用语义 token，必要时统一补充 token，而不是在页面内写散色值。
- [详情页状态较多，容易白屏] → 容器层统一状态映射，任何接口失败都回退到明确的说明卡，不让主流程中断。

## Migration Plan

1. 新增 ETF 估值工具页面、组件、API 模块与类型文件，不替换旧页面。
2. 在 `pages.config.ts` 或对应文件路由中注册新页面路径，确认分包归属仍在 ETF 子包内。
3. 先接静态/Mock 数据完成 UI 验证，再切换到真实 API。
4. 通过 lint 和手工验收验证首页、搜索结果页、详情页与状态页。
5. 如果上线后需要回滚，只移除新入口和新路由即可，旧 ETF 专业页不受影响。

## Open Questions

- 新工具入口挂在 ETF 子包首页、独立分享链接，还是外部运营入口直达，当前未在文档中完全定死。
- 搜索结果页是否需要支持热门词点击直达详情，还是先落到结果页做确认，实施时可根据路由设计决定。
- 参考站点中的图标素材是否需要完全一致，还是以项目现有图标体系做接近替代，实施阶段需最终确认。
