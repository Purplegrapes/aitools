## Context

`valuation-tool` 当前已经有首页、搜索页和详情页，但详情页只负责解释基金，不承担后续操作。现有 ETF 主工具页已经有 watchlist 增删能力，不过它依赖主列表 tab，且本次已明确不能复用现有 watchlist 接口，不符合“在 valuation-tool 内形成独立闭环”的目标。仓库中没有现成的“基金扫雷”页面，因此本次设计需要同时考虑：详情页固定底部栏、valuation-tool 专属自选接口、独立自选列表页、以及扫雷按钮的统一入口约定。

## Goals / Non-Goals

**Goals:**
- 在 valuation-tool 详情页增加固定底部操作栏，支持加入自选、自选列表、基金扫雷三个动作。
- 新建 valuation-tool 独立自选基金列表页，展示基金基础信息和当日涨幅。
- 详情页、自选列表页共享一套自选状态更新与详情跳转逻辑。
- 基金扫雷按钮形成稳定的路由入口，并透传当前基金代码。

**Non-Goals:**
- 不重构现有 ETF 主工具页的 watchlist tab，也不复用其 watchlist 接口。
- 不在本次变更中完整实现“基金扫雷”业务内容，只定义入口页/占位页和跳转协议。
- 不引入新的账户体系或新的收藏后端接口。

## Decisions

### 1. 在 valuation-tool 内新增独立自选页，而不是跳去 ETF 主页
- 方案 A：复用 ETF 主页 watchlist tab，详情页按钮只做跳转。
- 方案 B：在 valuation-tool 下新增独立自选页。
- 选择 B。这样交互路径保持在同一工具域内，用户不会从“基金解释器”被打断到另一个复杂页面，也更容易围绕当日涨幅做轻量列表展示。

### 2. 新增 valuation-tool 专属自选接口
- 方案 A：复用现有 ETF watchlist 接口。
- 方案 B：为 valuation-tool 新增专属接口，独立定义增删、自选列表与当日涨幅返回。
- 选择 B。需求已经明确不能复用现有接口，因此后端契约需要按 valuation-tool 的页面语义独立设计，避免被 ETF 主工具页的字段结构和交互历史绑定。

### 3. 当日涨幅由新接口直接支持，前端只做展示层兜底
- 方案 A：列表页自己再调用估值/行情接口逐个聚合涨幅。
- 方案 B：新自选列表接口直接返回或配套提供当日涨幅，前端只负责展示和少量降级。
- 选择 B。你已经明确新接口需要支持“列表中的当日涨幅”，因此涨幅能力应成为新契约的一部分，前端仅在字段缺失时做降级展示。

### 4. “基金扫雷”先提供独立入口页
- 方案 A：按钮先不做跳转，仅展示禁用态。
- 方案 B：新建 valuation-tool 的扫雷入口页或占位页，并以 `code` 作为入参约定后续能力。
- 选择 B。这样详情页的三个动作都是真正可点击的，同时不把完整扫雷业务耦合进本次变更。

## Risks / Trade-offs

- [新接口字段范围未及时对齐] → 在 spec 中明确最小字段集合：基金代码、名称、自选状态、当日涨幅。
- [新接口稳定性不足导致列表页不可用] → 页面保留空态/错误态与手动重试，不依赖现有 ETF watchlist 接口兜底。
- [详情页固定底栏遮挡正文] → 页面底部预留安全区与栏位高度，确保免责声明和最后一张卡片不会被覆盖。
- [基金扫雷能力尚未落地] → 本次只承诺入口与参数协议，页面文案明确为“入口页/即将开放”，避免误导为完整功能。

## Migration Plan

- 新增 valuation-tool 路由：自选列表页、基金扫雷入口页。
- 扩展 valuation-tool API 模块，新增专属自选增删、自选列表查询接口定义。
- 先完成接口契约接入与 mock/状态设计，再落详情页底部栏、自选列表页，最后补扫雷入口页与联调。
- 如果新接口首版未返回当日涨幅，则以前端降级态上线，但不回退到旧 watchlist 接口。

## Open Questions

- 新接口命名和路径约定是否统一落在 valuation-tool API 分组下，例如 `/api/v1/valuation-tool/watchlist/**`。
- 基金扫雷最终是独立页面还是外部能力嵌入；本次先按 valuation-tool 内部入口页设计。
