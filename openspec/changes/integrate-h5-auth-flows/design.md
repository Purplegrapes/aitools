## Context

当前项目已经把外部认证中转收口到 `auth` 网关，但 H5 场景下仍缺少独立的手机号验证码登录页。用户在 token 失效、首次访问受保护页面，或从内部链路被要求重新登录时，没有统一、稳定的本地 H5 登录入口。

已知约束：
- 项目要求 API 模块只返回 Alova `Method`，页面/组合式中通过 `useRequest` 调用。
- 现有 `auth` 模块已维护登录页、认证 API 和登录态写入工具，H5 手动登录应尽量复用这套契约。
- `etfUserStore` 维护业务登录态，后续页面鉴权已依赖统一 token 状态。

## Goals / Non-Goals

**Goals:**
- 新增 H5 手机号 + 短信验证码登录页，支持发送验证码与登录。
- 统一 H5 手动登录与现有认证链路的 token 写入、失败提示和跳转策略。
- 在 H5 登录成功后支持按 `referrer` 或默认首页回跳，避免登录后丢失用户上下文。

**Non-Goals:**
- 不改造外部小程序入口、中转网关或 tools 二跳链路。
- 不新增第三方登录方式（微信授权、账号密码等）。
- 不在本次范围内实现复杂风控/设备指纹策略。

## Decisions

### 1) 认证 API 复用 `auth` 模块现有能力并补齐 H5 登录调用

- 决策：H5 登录页复用 `auth` 侧认证 API 定义与统一登录态写入逻辑，不再围绕 `tamp` 中转页设计登录流程。
- 原因：认证能力已经收口在 `auth` 模块，继续在变更里绑定 `tamp` 会和现行主规范冲突。
- 备选方案：在 `tamp` 侧另起一套 API 和写入逻辑；短期可行，但会加剧重复实现。

### 2) 登录页聚焦手机号验证码两步流程

- 决策：登录页拆分“发送验证码”和“验证码登录”两个请求，分别维护 loading、错误提示和按钮禁用态。
- 原因：这符合现有 Alova `useRequest` 规范，也更利于控制防重复发送与提交态。
- 备选方案：把验证码发送与登录提交流程揉成一个操作；交互含义不清晰，错误也难定位。

### 3) 登录成功后复用统一认证写入动作

- 决策：手机号验证码登录成功后，复用统一认证写入动作更新 token、用户信息与持久化状态。
- 原因：保证 H5 手动登录与现有自动登录链路共享同一登录态契约，避免鉴权行为分叉。
- 备选方案：登录页自行 setToken；实现更快，但和现有登录态来源会脱节。

### 4) 回跳规则优先 `referrer`，无值时回首页

- 决策：登录成功后优先跳转 `referrer`，若 `referrer` 不存在或不合法，则回到默认首页。
- 原因：既能保留用户登录前的目标页意图，也能避免非法跳转。
- 备选方案：无论如何都回首页；实现简单，但会损失登录前上下文。

## Risks / Trade-offs

- [后端接口字段与预期不一致] → 在 API 层加响应适配与兜底错误映射，关键字段缺失时给出可观测错误文案。
- [验证码发送被频繁点击] → 登录页增加倒计时与按钮禁用状态，后端错误码映射为用户可理解提示。
- [登录成功后回跳目标不安全] → 继续沿用现有 `referrer` 校验规则，仅允许合法内部路径。
- [H5 手动登录与现有自动登录链路状态不一致] → 强制复用统一认证写入动作，并补最小回归验证。

## Migration Plan

1. 梳理并补齐 `auth` 侧 H5 登录需要的 API Method 与统一响应适配。
2. 复用现有登录态写入逻辑，确保手机号验证码登录成功后与既有鉴权状态一致。
3. 新增或完善 H5 登录页（手机号、验证码、发送倒计时、登录提交）。
4. 接入登录成功回跳与失败提示，覆盖 referrer 与默认首页两类落点。
5. 补最小回归验证，确认 H5 登录与现有受保护页面的鉴权契约一致。

## Open Questions

- 需后端最终确认手机号验证码接口路径与请求字段命名（例如 `mobile`/`phone`、`smsCode`/`code`）。
- 登录成功后默认回跳目标：严格优先 `referrer` 还是允许按业务页定制默认首页，需与产品确认。
- 验证码发送频率与失败提示文案是否有统一产品规范，需联调时确认。

## Integration Notes

- 当前实现假设发送验证码接口使用 `phone` 字段，验证码校验接口使用 `phone + code`。
- 当前实现假设验证码校验成功后先拿到授权码，再通过统一 token 交换接口获取正式登录态。
- H5 登录成功后即使 `user/me` 拉取失败，也不应回滚已成功写入的 token；联调时需要单独确认用户资料接口可用性。
- `referrer` 仍沿用内部路径白名单校验，联调环境应提供项目内路径而不是外部 URL。
