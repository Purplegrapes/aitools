## Context

当前 `src/subPages/tamp/index.vue` 仍处于“临时从 URL 读取 token”阶段，`code` 换 token 的 API 调用逻辑被注释，未形成正式登录同步流程。与此同时，项目缺少独立的 H5 手机号验证码登录页，导致 H5 来源用户在 token 失效或首次访问时没有统一可用的登录入口。另一个问题是：外部小程序进入 `subPages` 还没有统一前置入口约束，导致部分页面可能绕开中转页。

已知约束：
- 项目要求 API 模块只返回 Alova `Method`，页面/组合式中通过 `useRequest` 调用。
- 现有 `tampStore` 负责记录外部来源信息与回跳参数，`etfUserStore` 维护业务登录态。
- 外部来源跳转规则（小程序/H5/内部）已在 TAMP 模块中存在，需要保持兼容。

## Goals / Non-Goals

**Goals:**
- 在 TAMP 中转页实现“进入即尝试 `code` 换 `token`”并同步登录态。
- 将 TAMP 中转页建设为外部小程序访问 `subPages` 的统一入口网关，确保先中转后落页。
- 新增 H5 手机号 + 短信验证码登录页，支持发送验证码与登录。
- 统一中转登录与手动登录后的 token 写入、失败提示和跳转策略。
- 在 token 获取失败时提供可恢复路径（跳登录页或来源回退），避免中转页卡死。

**Non-Goals:**
- 不改造后端认证协议或 token 结构。
- 不新增第三方登录方式（微信授权、账号密码等）。
- 不在本次范围内实现复杂风控/设备指纹策略。

## Decisions

### 1) 中转页优先走 `code` 换 token，保留 token 直传兼容

- 决策：外部访问时优先检测 `query.code` 并调用 `tokenByCode`；若无 `code` 且带 `token`，继续兼容 token 直传方案。
- 原因：满足“进入中转页直接 code 换 token”目标，同时避免历史链接立即失效。
- 备选方案：仅支持 `code`，移除 token 兼容；该方案风险较高，会中断存量跳转链路，暂不采用。

### 2) 外部小程序访问 subPages 强制先经过中转页

- 决策：定义统一跳转规则：外部小程序若目标是 `subPages/**`，必须先跳到 `subPages/tamp/index`，通过 `referer` 携带目标路径并完成鉴权后再重定向。
- 原因：统一入口可保证认证、来源识别、参数清洗和回退逻辑在一个点治理，避免各页面重复实现“半套中转”逻辑。
- 备选方案：每个目标页自行做鉴权拦截；维护成本高且容易产生行为不一致。

### 3) Tools 页面采用“中转页 + tools entry”双层网关

- 决策：当目标路径匹配 `subPages/tools/**` 时，中转页不直接落目标业务页，而是二跳到 `subPages/tools/entry`，并透传 `targetUrl/ticket/timestamp/nonce/sign/traceId` 等 tools-embed 参数。
- 原因：`tools/entry` 已承载换票建会话与 bridge 协议能力，直接落工具页会破坏既有嵌入式安全边界。
- 备选方案：将 tools 换票能力并入 tamp；短期改动面大、回归成本高，不作为当前迭代目标。

### 4) 登录态写入由统一认证动作封装

- 决策：在 TAMP 流程与 H5 登录页都通过同一认证写入动作（封装到 store action 或 composable）更新 token、登录标记与必要用户信息。
- 原因：避免两套登录入口写入逻辑分叉，减少“页面认为已登录但 store 未同步”的状态不一致。
- 备选方案：各页面分别 setToken；实现更快，但长期维护风险高。

### 5) 新增 H5 手机验证码登录页并显式区分“发送验证码/提交登录”

- 决策：登录页拆分两个请求：发送验证码与验证码登录，均用 `useRequest` 驱动 loading/error；验证码按钮带倒计时与重复发送限制。
- 原因：契合现有 Alova 使用规范，并提升交互可预期性。
- 备选方案：提交登录时隐式触发发送验证码；用户体验与错误定位都更差。

### 6) 失败分流：外部来源优先回来源，内部来源跳本地登录

- 决策：
  - 中转登录失败时，若存在 `loginUrl` 且来源为 H5，跳外部 `loginUrl`；
  - 小程序来源走 `handleExternalRedirect`；
  - 内部/H5 无来源参数则跳本地 H5 登录页。
- 原因：与现有来源识别模型一致，降低外部接入改造成本。
- 备选方案：失败统一跳本地登录页；会破坏外部集成方预期。

## Risks / Trade-offs

- [后端接口字段与预期不一致] → 在 API 层加响应适配与兜底错误映射，关键字段缺失时给出可观测错误文案。
- [并发进入中转页触发重复登录] → 中转页增加单次执行锁（in-flight guard）避免重复提交。
- [验证码发送被频繁点击] → 登录页增加倒计时与按钮禁用状态，后端错误码映射为用户可理解提示。
- [兼容历史 token 直传导致行为分叉] → 在中转页明确优先级（`code` > `token`）并记录日志，后续可逐步下线 token 直传。
- [外部接入方继续直跳目标 subPages] → 提供统一跳转 URL 约定与接入说明，并在目标页加入兜底检测（发现外部来源但缺失中转上下文时回到中转页）。
- [tools 目标页跳过 tools/entry] → 在 tamp 增加路径分支保护：命中 `subPages/tools/**` 时强制转发到 tools/entry 而非直接放行。

## Migration Plan

1. 扩展 `src/subPages/tamp/api`：补齐 `code` 换 token、验证码发送、验证码登录 Method。
2. 建立外部小程序到 `subPages` 的统一跳转约定与路径构造工具，默认先指向 TAMP 中转页。
3. 新增统一认证写入逻辑（复用 `etfUserStore` 或新增认证 composable），供中转页与登录页共用。
4. 改造 `src/subPages/tamp/index.vue`：接入 `code` 自动登录、失败分流与成功跳转。
5. 新增 H5 登录页并接入路由（手机号、验证码、发送倒计时、登录提交）。
6. 在 TAMP 增加 tools 目标分支转发，确保通过 `subPages/tools/entry` 建立 tools 会话。
7. 联调来源回跳逻辑，验证 H5/小程序/内部三类入口以及“外部访问 subPages 必经中转页”约束。
8. 增补最小回归测试清单（中转成功、登录成功、失败回退、验证码节流、tools 二跳成功）。

## Open Questions

- 需后端最终确认手机号验证码接口路径与请求字段命名（例如 `mobile`/`phone`、`smsCode`/`code`）。
- `code` 换 token 返回体是否包含用户基础信息；若不包含，是否需要追加拉取用户信息接口。
- H5 登录成功后默认回跳目标：优先 `referer` 还是固定首页，需与产品确认。
- TAMP 转发到 tools entry 时，登录同步与 embed 换票的先后策略是否需要按来源做差异化（例如仅小程序来源要求先换票）。
